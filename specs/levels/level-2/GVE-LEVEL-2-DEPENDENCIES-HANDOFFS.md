## Summary

GVE Level 2 Dependencies and Handoffs defines deterministic operation-dependency and sequencing graphs, pre-execution data-handoff declarations, runtime validation of actual handoffs before dependent-operation attempt, blocked and unattempted operation behavior, deterministic propagation, and partial workflow progress. It preserves the normative distinction between dependencies, declared handoff contracts, and actual runtime data handoffs and preserves all inherited effect-state truthfulness.

## Definitions

### Level 2 operation dependency (`L2-DH-OPERATION-DEPENDENCY`)

An application-independent prerequisite relationship in which one governed operation may become eligible for attempt only after the declared prerequisite operation satisfies the dependency condition established by the accepted workflow plan.

### Sequencing edge (`L2-DH-SEQUENCING-EDGE`)

A directed composition relationship from one operation identity to another that constrains eligible execution order without by itself declaring a data transfer.

### Dependency endpoint (`L2-DH-DEPENDENCY-ENDPOINT`)

A source or target operation identity named by a dependency or sequencing edge and required to resolve uniquely to an operation in the same accepted workflow plan.

### Dependency graph (`L2-DH-DEPENDENCY-GRAPH`)

The directed acyclic graph formed by the accepted workflow plan's operation identities and explicit dependency and sequencing edges.

### Data-handoff declaration (`L2-DH-HANDOFF-DECLARATION`)

The application-independent pre-execution contract that names exactly one source operation, one target operation, the required source result, the intended target input, compatibility conditions, authority requirements, and evidence obligations.

### Actual data handoff (`L2-DH-ACTUAL-HANDOFF`)

The runtime transfer candidate produced from actual evidence-supported source result data for a target operation under exactly one previously accepted handoff declaration.

### Source-result obligation (`L2-DH-SOURCE-RESULT-OBLIGATION`)

The requirement that the source operation produce the specific declared result data with sufficient evidence and authority before that data may participate in an actual handoff.

### Target-input obligation (`L2-DH-TARGET-INPUT-OBLIGATION`)

The requirement that the target operation and its selected plugin accept the declared application-independent input role before an actual handoff may satisfy it.

### Handoff compatibility (`L2-DH-HANDOFF-COMPATIBILITY`)

The application-independent determination that actual source result data satisfies the accepted declaration's target-input, identity, version, authority, and evidence conditions without the core interpreting plugin-specific data meaning.

### Handoff authority (`L2-DH-HANDOFF-AUTHORITY`)

The authority required for the source result to be disclosed and for the target operation to consume it under the accepted workflow plan.

### Handoff evidence (`L2-DH-HANDOFF-EVIDENCE`)

The evidence sufficient to establish the source result's identity, provenance, availability, integrity, declared compatibility, and authorized transfer to the target operation.

### Blocked operation (`L2-DH-BLOCKED-OPERATION`)

An operation that remains unattempted because a required dependency condition or actual handoff validation condition is unsatisfied, unavailable, invalid, unauthorized, or insufficiently evidenced.

### Skipped operation (`L2-DH-SKIPPED-OPERATION`)

An operation that remains unattempted because an accepted workflow rule explicitly selects non-attempt after a preceding outcome. Skipped is distinct from blocked and failed.

### Failed operation (`L2-DH-FAILED-OPERATION`)

An operation for which an authorized attempt occurred and the governed operation did not complete successfully. Failure must not be assigned to an operation that was never attempted.

### Unattempted operation (`L2-DH-UNATTEMPTED-OPERATION`)

An operation for which no governed attempt occurred. Blocked and skipped operations are unattempted, but their distinct causes and statuses must remain explicit.

### Dependency and handoff propagation (`L2-DH-PROPAGATION`)

The deterministic derivation of downstream eligibility, blocked, skipped, or unattempted status from accepted dependency, sequencing, failure, and handoff outcomes without converting those statuses into effect claims.

### Partial workflow progress (`L2-DH-PARTIAL-PROGRESS`)

A workflow condition in which some operations have valid evidence-supported outcomes while other operations remain failed, blocked, skipped, or unattempted. Partial progress does not imply complete workflow success.

## Normative requirements

### L2-DH-REQ-001

Every dependency and sequencing edge must be explicit and must name exactly one source operation identity and one target operation identity in the same accepted workflow plan.

References: `L2-DH-OPERATION-DEPENDENCY`, `L2-DH-SEQUENCING-EDGE`, `L2-DH-DEPENDENCY-ENDPOINT`

### L2-DH-REQ-002

Every dependency endpoint must resolve deterministically and uniquely to an operation in the accepted workflow plan. Missing, duplicate, cross-workflow, self-referential, ambiguous, or otherwise unresolved endpoints must fail closed.

References: `L2-DH-DEPENDENCY-ENDPOINT`, `L2-WC-OPERATION-IDENTITY`, `L2-WC-COMPLETE-PLAN`

### L2-DH-REQ-003

The dependency graph must be acyclic. Any direct or transitive cycle among dependency or sequencing edges must fail workflow-plan acceptance before execution.

References: `L2-DH-DEPENDENCY-GRAPH`, `L2-WC-PLAN-ACCEPTANCE`

### L2-DH-REQ-004

Operation dependencies and sequencing edges must remain distinct from data-handoff declarations and actual data handoffs. A dependency may exist without a handoff, and a handoff declaration must not be inferred solely from a dependency.

References: `L2-DH-OPERATION-DEPENDENCY`, `L2-DH-HANDOFF-DECLARATION`, `L2-DH-ACTUAL-HANDOFF`

### L2-DH-REQ-005

Every planned data handoff must be represented by exactly one valid pre-execution handoff declaration accepted as part of the complete workflow plan.

References: `L2-DH-HANDOFF-DECLARATION`, `L2-WC-COMPLETE-PLAN`, `L2-WC-PLAN-ACCEPTANCE`

### L2-DH-REQ-006

Each handoff declaration must identify one source operation, one target operation, the required source result, the intended target input, compatibility conditions, required authority, and required evidence.

References: `L2-DH-HANDOFF-DECLARATION`, `L2-DH-SOURCE-RESULT-OBLIGATION`, `L2-DH-TARGET-INPUT-OBLIGATION`, `L2-DH-HANDOFF-COMPATIBILITY`, `L2-DH-HANDOFF-AUTHORITY`, `L2-DH-HANDOFF-EVIDENCE`

### L2-DH-REQ-007

A missing, malformed, conflicting, ambiguous, unauthorized, incomplete, or non-uniquely attributable handoff declaration must fail workflow-plan acceptance before any operation begins.

References: `L2-DH-HANDOFF-DECLARATION`, `L2-WC-PLAN-ACCEPTANCE`

### L2-DH-REQ-008

An actual data handoff must not occur unless it is attributable to exactly one accepted handoff declaration and exactly one source and target operation pair.

References: `L2-DH-ACTUAL-HANDOFF`, `L2-DH-HANDOFF-DECLARATION`

### L2-DH-REQ-009

Before a target operation may be attempted, the core must validate every required actual handoff for that operation against its accepted declaration, source result, target input, compatibility conditions, authority, and evidence obligations.

References: `L2-DH-ACTUAL-HANDOFF`, `L2-DH-SOURCE-RESULT-OBLIGATION`, `L2-DH-TARGET-INPUT-OBLIGATION`, `L2-DH-HANDOFF-COMPATIBILITY`, `L2-DH-HANDOFF-AUTHORITY`, `L2-DH-HANDOFF-EVIDENCE`

### L2-DH-REQ-010

An undeclared, missing, unavailable, invalid, incompatible, unauthorized, ambiguous, stale, or insufficiently evidenced actual handoff must prevent the dependent operation from being attempted.

References: `L2-DH-ACTUAL-HANDOFF`, `L2-DH-BLOCKED-OPERATION`, `L2-DH-UNATTEMPTED-OPERATION`

### L2-DH-REQ-011

A dependent operation whose required dependency condition is unsatisfied or whose required actual handoff is unavailable or invalid must be recorded as blocked and unattempted unless an accepted workflow rule deterministically assigns a distinct skipped status.

References: `L2-DH-BLOCKED-OPERATION`, `L2-DH-SKIPPED-OPERATION`, `L2-DH-UNATTEMPTED-OPERATION`

### L2-DH-REQ-012

Blocked, skipped, failed, and unattempted statuses must remain distinct. A blocked or skipped operation must not be reported as failed because no attempt occurred, and a failed operation must have evidence of an authorized attempt.

References: `L2-DH-BLOCKED-OPERATION`, `L2-DH-SKIPPED-OPERATION`, `L2-DH-FAILED-OPERATION`, `L2-DH-UNATTEMPTED-OPERATION`

### L2-DH-REQ-013

Dependency and handoff outcomes must propagate deterministically through the accepted dependency graph. Propagation must preserve the specific cause of each downstream blocked, skipped, or unattempted status.

References: `L2-DH-DEPENDENCY-GRAPH`, `L2-DH-PROPAGATION`

### L2-DH-REQ-014

Dependency, sequencing, declaration, handoff, blocked, skipped, failed, unattempted, and propagation statuses must not by themselves establish that any requested effect was attempted, completed, observed, or verified.

References: `L2-DH-PROPAGATION`, `LEVEL-2-COMPOSITION-LAYER`

### L2-DH-REQ-015

A workflow may record partial progress only when each operation's actual status and evidence are retained independently. Outcomes of completed or failed operations must not be erased by downstream blocking, and partial progress must not be represented as complete workflow success.

References: `L2-DH-PARTIAL-PROGRESS`, `L2-DH-PROPAGATION`

### L2-DH-REQ-016

The core may validate application-independent handoff identity, attribution, compatibility claims, authority, and evidence obligations, but it must not interpret plugin-specific source-result or target-input meaning.

References: `L2-DH-HANDOFF-COMPATIBILITY`, `L2-WC-PLUGIN-MEANING-BOUNDARY`

### L2-DH-REQ-017

This specification must not define concrete handoff serialization, plugin-specific data schemas, retries, resume, compensation, rollback, transactions, maintained runtime implementation, evidence aggregation, or authoritative result assembly.

References: `LEVEL-2-RESULT-ASSEMBLY`, `LEVEL-2-LOCAL-PLUGIN-BOUNDARIES`
