## Summary

GVE Level 2 Local Plugin Boundaries defines distinct composition-facing responsibilities for local filesystem modification, command execution, and local Git operation plugin domains. It requires deterministic unique domain assignment, explicit validated-contract, evidence, handoff, and result obligations, application-independent core behavior, plugin-owned domain-specific interpretation and execution, and bounded follow-on plugin specification work without authorizing concrete production payload schemas or maintained implementations.

## Definitions

### Local filesystem modification domain (`L2-LPB-FILESYSTEM-DOMAIN`)

The plugin domain responsible for interpreting, validating, executing, evidencing, and reporting governed changes to local filesystem objects and metadata, excluding command process semantics and Git repository semantics that belong to other domains.

### Command-execution domain (`L2-LPB-COMMAND-DOMAIN`)

The plugin domain responsible for interpreting, validating, executing, evidencing, and reporting governed local process invocation and process outcome semantics, excluding direct filesystem-object mutation semantics and Git repository semantics owned by other domains.

### Local Git operations domain (`L2-LPB-LOCAL-GIT-DOMAIN`)

The plugin domain responsible for interpreting, validating, executing, evidencing, and reporting governed operations on local Git repository state, excluding remote-service and hosting-provider behavior.

### Domain operation ownership (`L2-LPB-DOMAIN-OWNERSHIP`)

The normative assignment of one operation's domain-specific meaning to exactly one plugin domain according to non-overlapping composition-facing responsibility boundaries.

### Deterministic domain assignment (`L2-LPB-DETERMINISTIC-ASSIGNMENT`)

The application-independent determination that one declared operation maps uniquely to one eligible plugin domain without ambiguity, overlap, fallback interpretation, or core inference of domain-specific meaning.

### Validated operation-contract exposure (`L2-LPB-CONTRACT-EXPOSURE`)

The domain obligation to expose sufficient application-independent contract identity, authority, input, output, evidence, handoff, and result obligations for core validation and composition while retaining domain-specific meaning inside the selected plugin.

### Domain evidence obligation (`L2-LPB-DOMAIN-EVIDENCE`)

The obligation of each selected plugin domain to produce attributable evidence sufficient for its operation-level claims without requiring the core to interpret domain-specific evidence meaning.

### Domain handoff obligation (`L2-LPB-DOMAIN-HANDOFF`)

The obligation of each selected plugin domain to declare and expose application-independent handoff roles and compatibility claims while retaining interpretation of domain-specific source-result and target-input meaning.

### Domain result obligation (`L2-LPB-DOMAIN-RESULT`)

The obligation of each selected plugin domain to produce an attributable composition-facing operation result with distinct effect-state claims and supporting evidence while retaining plugin-specific result meaning.

### Core and domain interpretation boundary (`L2-LPB-CORE-DOMAIN-BOUNDARY`)

The boundary under which the core owns application-independent identity, authority, composition, dependency, handoff, evidence-sufficiency, and result-assembly rules, while each plugin owns domain-specific interpretation, validation, execution, evidence meaning, and result meaning.

### Domain non-overlap (`L2-LPB-DOMAIN-NONOVERLAP`)

The requirement that filesystem, command-execution, and local-Git composition-facing responsibilities do not overlap in a way that permits more than one domain to claim ownership of the same operation meaning.

### Follow-on plugin specification work (`L2-LPB-FOLLOWON-WORK`)

Separately governed subordinate work that defines plugin-specific contracts and semantics for the filesystem, command-execution, and local-Git domains after these common composition boundaries are accepted.

## Normative requirements

### L2-LPB-REQ-001

The local filesystem modification, command-execution, and local Git operation domains must have distinct composition-facing responsibilities sufficient for deterministic unique plugin assignment.

References: `L2-LPB-FILESYSTEM-DOMAIN`, `L2-LPB-COMMAND-DOMAIN`, `L2-LPB-LOCAL-GIT-DOMAIN`, `L2-LPB-DETERMINISTIC-ASSIGNMENT`

### L2-LPB-REQ-002

Each declared operation requiring local plugin execution must be attributable to exactly one eligible plugin domain before workflow-plan acceptance. Missing, duplicate, ambiguous, or overlapping assignment must fail closed.

References: `L2-LPB-DOMAIN-OWNERSHIP`, `L2-LPB-DETERMINISTIC-ASSIGNMENT`, `L2-WC-PLUGIN-BINDING`, `L2-WC-PLAN-ACCEPTANCE`

### L2-LPB-REQ-003

The local filesystem modification domain owns governed meaning for direct local filesystem-object and metadata changes and must not claim ownership solely because another domain's execution may incidentally read or write filesystem state.

References: `L2-LPB-FILESYSTEM-DOMAIN`, `L2-LPB-DOMAIN-NONOVERLAP`

### L2-LPB-REQ-004

The command-execution domain owns governed meaning for local process invocation and process outcome and must not become the default owner for operations whose governed meaning belongs to filesystem or Git semantics.

References: `L2-LPB-COMMAND-DOMAIN`, `L2-LPB-DOMAIN-NONOVERLAP`

### L2-LPB-REQ-005

The local Git operations domain owns governed meaning for local Git repository state and history operations and must exclude GitHub, hosting-provider, credential, remote publication, issue, pull-request, and review behavior.

References: `L2-LPB-LOCAL-GIT-DOMAIN`, `L2-LPB-DOMAIN-NONOVERLAP`

### L2-LPB-REQ-006

Every plugin domain must expose a validated operation contract sufficient for application-independent core validation of contract identity, selected-plugin binding, authority, declared inputs and outputs, evidence obligations, handoff roles, and result obligations.

References: `L2-LPB-CONTRACT-EXPOSURE`, `L2-WC-CONTRACT-ATTRIBUTION`, `L2-WC-CORE-CONTRACT-FACTS`

### L2-LPB-REQ-007

Every selected plugin domain must produce operation evidence attributable to the operation identity, selected plugin, validated contract, authority, attempt, outcome, and effect-state claims it supports.

References: `L2-LPB-DOMAIN-EVIDENCE`, `L2-RA-OPERATION-EVIDENCE`, `L2-RA-RESULT-TRACEABILITY`

### L2-LPB-REQ-008

Every selected plugin domain participating in a handoff must expose application-independent declaration and compatibility information sufficient for core validation while retaining interpretation of domain-specific source-result and target-input meaning.

References: `L2-LPB-DOMAIN-HANDOFF`, `L2-DH-HANDOFF-DECLARATION`, `L2-DH-HANDOFF-COMPATIBILITY`

### L2-LPB-REQ-009

Every selected plugin domain must produce one composition-facing operation result attributable to the operation identity, selected plugin, validated contract, lifecycle outcome, separate effect-state claims, and supporting evidence.

References: `L2-LPB-DOMAIN-RESULT`, `L2-RA-OPERATION-RESULT`, `L2-RA-OPERATION-OUTCOME`

### L2-LPB-REQ-010

The core must remain application-independent and must not interpret, derive, normalize, override, or validate domain-specific operation, input, output, evidence, handoff, or result meaning.

References: `L2-LPB-CORE-DOMAIN-BOUNDARY`, `L2-WC-PLUGIN-MEANING-BOUNDARY`

### L2-LPB-REQ-011

Each plugin domain must own domain-specific interpretation, validation, execution, evidence meaning, handoff meaning, and result meaning for operations assigned to that domain.

References: `L2-LPB-CORE-DOMAIN-BOUNDARY`, `L2-LPB-DOMAIN-OWNERSHIP`

### L2-LPB-REQ-012

The core may validate application-independent identity, authority, contract attribution, dependency, declaration, evidence-sufficiency, and result-assembly obligations exposed by a plugin without assuming domain-specific semantics.

References: `L2-LPB-CORE-DOMAIN-BOUNDARY`, `L2-WC-OPERATION-IDENTITY`, `L2-DH-HANDOFF-DECLARATION`, `L2-RA-EVIDENCE-AGGREGATION`

### L2-LPB-REQ-013

Filesystem, command-execution, and local-Git domain rules must be non-overlapping. A rule set that permits two or more domains to claim the same governed operation meaning must fail specification validation closed.

References: `L2-LPB-DOMAIN-NONOVERLAP`, `L2-LPB-DETERMINISTIC-ASSIGNMENT`

### L2-LPB-REQ-014

Incidental implementation effects must not determine domain ownership. Domain assignment must follow the operation's governed meaning and validated contract rather than incidental filesystem access, process use, or repository access.

References: `L2-LPB-DOMAIN-OWNERSHIP`, `L2-LPB-DOMAIN-NONOVERLAP`

### L2-LPB-REQ-015

Follow-on plugin-specific specification work must proceed only after acceptance of these boundaries and must be ordered as filesystem plugin specification, command-execution plugin specification, and local-Git plugin specification unless a later governed dependency analysis establishes a stricter order.

References: `L2-LPB-FOLLOWON-WORK`

### L2-LPB-REQ-016

Each follow-on plugin-specific specification must import this document and the applicable Level 2 composition, dependency, handoff, and result-assembly authorities and must remain bounded to one plugin domain.

References: `L2-LPB-FOLLOWON-WORK`, `L2-DA-IMPORT`, `LEVEL-2-WORKFLOW-COMPOSITION`, `LEVEL-2-DEPENDENCIES-HANDOFFS`, `LEVEL-2-RESULT-ASSEMBLY`

### L2-LPB-REQ-017

This specification must not define concrete production payload schemas, exhaustive operation catalogs, maintained plugin implementations, remote-service behavior, credentials, publication semantics, or bootstrap executor operation formats.

References: `LEVEL-2-LOCAL-PLUGIN-BOUNDARIES`
